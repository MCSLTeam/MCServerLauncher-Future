# GitHub Release Workflow Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a manual Windows-hosted GitHub release workflow that publishes WPF and daemon packages for Windows runtime identifiers, publishes daemon packages for Linux and macOS runtime identifiers, archives Windows packages as `.zip` and Linux/macOS packages as `.tar.gz`, and publishes a normal release or pre-release using `Release.md` as the release body.

**Architecture:** One `workflow_dispatch` workflow runs on `windows-latest`, fans out across Windows, Linux, and macOS runtime identifiers, and uses workflow inputs to include or exclude PDBs. Each matrix job publishes daemon output for every runtime, publishes WPF output only for Windows runtimes, writes a metadata file for each package mode, and uploads a single archive plus metadata per runtime. A final release step aggregates all assets and creates the GitHub Release body from `Release.md` plus a generated package table and asset list.

**Tech Stack:** GitHub Actions, PowerShell 7, .NET 10, Windows hosted runners, `actions/upload-artifact`, `actions/download-artifact`, and `softprops/action-gh-release` or an equivalent release action.

### Task 1: Add the release note source file

**Files:**
- Create: `Release.md`

- [ ] **Step 1: Write the release note template**

```md
# MCServerLauncher Future Release

This release packages the Windows, Linux, and macOS outputs produced by the release workflow.

## Package Rules

- Windows packages use `.zip` archives.
- Linux and macOS packages use `.tar.gz` archives.
- Each runtime is published twice: self-contained and framework-dependent.
- Windows packages contain both the WPF client and the daemon.
- Linux and macOS packages contain the daemon only.

## Notes

- Use the GitHub Release body generated by the release workflow for the exact package matrix and uploaded assets.
```

- [ ] **Step 2: Keep the file generic enough for repeated releases**

Do not hard-code version numbers, commit hashes, or runtime identifiers in the file. The workflow appends those run-specific details at release time.

### Task 2: Add the Windows release workflow

**Files:**
- Create: `.github/workflows/windows-release.yml`

- [ ] **Step 1: Define manual workflow inputs**

```yml
on:
  workflow_dispatch:
    inputs:
      tag_name:
        description: Git tag to publish
        required: true
        type: string
      prerelease:
        description: Mark the GitHub Release as a pre-release
        required: true
        type: boolean
        default: false
      include_pdb:
        description: Include PDB files in the publish output
        required: true
        type: boolean
        default: false
```

- [ ] **Step 2: Build both projects for each Windows runtime identifier**

```yml
strategy:
  matrix:
    include:
      - rid: win-x64
        family: windows
        architecture: x64
        include_wpf: true
        archive_ext: zip
      - rid: win-x86
        family: windows
        architecture: x86
        include_wpf: true
        archive_ext: zip
      - rid: win-arm64
        family: windows
        architecture: arm64
        include_wpf: true
        archive_ext: zip
      - rid: linux-x64
        family: linux
        architecture: x64
        include_wpf: false
        archive_ext: tar.gz
      - rid: linux-arm64
        family: linux
        architecture: arm64
        include_wpf: false
        archive_ext: tar.gz
      - rid: osx-x64
        family: macos
        architecture: x64
        include_wpf: false
        archive_ext: tar.gz
      - rid: osx-arm64
        family: macos
        architecture: arm64
        include_wpf: false
        archive_ext: tar.gz
```

- [ ] **Step 3: Publish and package each matrix item**

```pwsh
$publishArgs = @(
  'publish',
  $projectPath,
  '-c', 'Release',
  '-r', $env:RID,
  '--self-contained', $selfContained,
  '-o', $publishDir
)
```

Create one archive per package mode for each runtime. Windows archives use `.zip`; Linux and macOS archives use `.tar.gz`. Windows packages include both WPF and daemon outputs, while Linux/macOS packages include the daemon only.

- [ ] **Step 4: Create the release from `Release.md`**

```yml
- name: Read release body
  shell: pwsh
  run: |
    $body = Get-Content Release.md -Raw
    $body += "`n## Published Packages`n"
    $body += "`n| RID | Mode | Archive | Contents |`n| --- | --- | --- | --- |`n"
    $body += "`n- Release tag: ${{ inputs.tag_name }}`n"
    $body += "- Pre-release: ${{ inputs.prerelease }}`n"
    $body += "- Include PDB: ${{ inputs.include_pdb }}`n"
    Set-Content -Path release-body.md -Value $body -Encoding utf8
```

  Use the generated body file in the release action and attach all archived artifacts.
